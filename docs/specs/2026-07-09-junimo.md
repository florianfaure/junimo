# Junimo — spec de design

**Date** : 2026-07-09 · **Statut** : validée par Florian (brainstorming du 2026-07-09)

## Contexte et objectif

Quand on travaille avec Claude Code sur un abonnement Pro/Max, les informations vitales (jauges d'utilisation, MCPs connectés, modèle, plan) ne sont visibles qu'en pleine session via `/usage` ou en fouillant des fichiers de config. Junimo est une app **menu bar macOS** : une icône pixel-art dans la barre de menu ouvre un **overlay dashboard** qui montre tout d'un coup d'œil, avec une UI tech teintée de Stardew Valley.

Utilisateur cible v1 : Florian, en local, usage personnel. Pas de distribution signée ni de multi-comptes en v1.

## Décisions structurantes (et alternatives écartées)

| Décision | Choix | Alternatives écartées |
|---|---|---|
| Stack | **Tauri v2** (Rust + webview) | Electron (trop lourd pour un overlay) ; SwiftUI natif (pixel art laborieux, itérations lentes) |
| Source des données | **100 % local, lecture seule** des fichiers de Claude Code | Réutilisation du token OAuth de Claude Code : **écartée, contraire aux ToS Anthropic (politique de février 2026)** ; login OAuth intégré : même problème + complexité |
| Jauges | **Estimations locales** calculées depuis les transcripts, étiquetées « estimé », plafonds calibrables | Pourcentages officiels de `/usage` : aucun endpoint public pour Pro/Max |
| Frontend | Vite + TypeScript vanilla | React/Svelte : inutile pour un seul écran |
| Gestion de projet | Roadmapped (`docs/tasks/`, CLI) | — (demande explicite) |

## Sources de données (toutes vérifiées sur la machine cible)

Lecture seule, jamais d'écriture dans `~/.claude*`. Chemins résolus depuis `$HOME`.

1. **`~/.claude/projects/**/*.jsonl`** — transcripts : chaque ligne assistant porte `timestamp`, `message.model`, `message.usage` (input/output/cache tokens). Sert au calcul des jauges et de l'activité.
2. **`~/.claude.json`** — `oauthAccount` (displayName, emailAddress, organizationName, organizationType, userRateLimitTier, billingType), `mcpServers` (scope global), `projects.<path>.mcpServers` (scope projet).
3. **`~/.claude/settings.json`** — `model` (modèle par défaut), `enabledPlugins`.
4. **`claude --version`** — version de Claude Code (exécuté au refresh, avec timeout).

## Architecture

```
src-tauri/ (Rust)
├── collector/            # cœur métier, testé unitairement
│   ├── transcripts.rs    # scan + parse en flux des JSONL (fichiers avec mtime < 8 jours)
│   ├── windows.rs        # fenêtres glissantes : 5h courante, 7 jours, 7 jours par famille de modèle
│   ├── config.rs         # lecture ~/.claude.json, settings.json, version CLI
│   └── mod.rs            # Snapshot = struct sérialisée unique envoyée au front
├── tray.rs               # icône template pixel-art, clic → toggle de l'overlay
└── lib.rs                # commandes Tauri : get_snapshot(), get_settings(), set_settings()

src/ (Vite + TS vanilla)
├── main.ts               # invoke get_snapshot au montage + toutes les 30 s + à chaque ouverture
├── ui/                   # rendu des 3 sections (jauges, MCPs, compte)
└── styles/               # thème pixel Stardew
```

- **Snapshot** : le backend produit un unique objet JSON (jauges, MCPs, compte, méta). Le front est un pur affichage — aucune logique métier côté TS.
- **Overlay** : fenêtre frameless, toujours au-dessus, ancrée sous l'icône tray (`tauri-plugin-positioner`), fermée au blur. Pas d'icône Dock (app "accessory").
- **Erreurs** : chaque source est indépendante ; si l'une échoue (fichier absent, JSON invalide, CLI indisponible), sa section affiche un état dégradé explicite, les autres vivent. Jamais de crash du collecteur sur une ligne JSONL malformée (ligne ignorée, comptée dans les métriques de parse).

## Calcul des jauges (le cœur à risque — TDD)

- **Événements** : lignes de type assistant portant `message.usage` ; déduplication par `(message.id, requestId)` pour ne pas compter deux fois les événements dupliqués entre fichiers (resume de session).
- **Pondération** : total tokens = input + output + cache_creation + cache_read. Le poids relatif du cache et le mapping tokens→coût sont des constantes nommées, documentées comme approximations.
- **Fenêtre 5h** : le bloc courant démarre au premier message suivant l'expiration du bloc précédent (heure arrondie), comme le fait Anthropic ; affiche consommation, % du plafond estimé, heure de reset.
- **Fenêtre 7 jours** : fenêtre **fixe sur une grille de 7 jours** projetée sur une référence de reset : le réglage `weekly_reset_reference` (recopié une fois depuis `/usage`, prioritaire), sinon une estimation chaînée depuis l'historique local (fenêtres démarrant au minuit local du premier événement après expiration de la précédente), sinon repli glissant. Global + par famille de modèle (Fable/Opus vs Sonnet/Haiku). *(Amendé le 2026-07-10 après vérification #10 : le glissant initial puis l'ancrage sur `subscriptionCreatedAt` divergeaient des resets réels de `/usage`.)*
- **Bloc 5h** : démarre à la **demi-heure inférieure** du premier message du bloc (calibré sur `/usage`, granularité 30 min observée).
- **Calibration par jauge** : aucune pondération de tokens ne rend les 3 compteurs `/usage` cohérents entre eux (constat 2026-07-10) — chaque jauge a donc son propre plafond, calibré indépendamment sur `/usage` et ajustable dans `junimo-settings.json`. Les pourcentages restent des estimations, fiables surtout à mix d'usage constant.
- **Plafonds** : constantes par défaut selon le plan détecté (billingType/rateLimitTier), ajustables dans les réglages de l'app. Les jauges affichent toujours la mention « estimé ».
- **Performance** : ne parser que les fichiers `mtime < 8 jours`, en flux (pas de chargement mémoire complet). Budget : rafraîchissement < 500 ms sur un historique réel.

## UI — « tech d'abord, pixel en accent »

- Panneau ~360×540, fond brun/parchemin sombre inspiré des menus Stardew, bordures 9-slice pixel.
- **Jauges** : barres d'énergie façon Stardew (segments pixel, vert → orange → rouge), heure de reset et valeurs chiffrées en police mono lisible.
- **Titres et labels** : police pixel embarquée localement (pas de CDN). **Données** : police mono système.
- Sprites : junimo animé (idle) près du titre, icônes pixel pour MCPs/compte. `image-rendering: pixelated`.
- Icône tray : junimo monochrome (template image, s'adapte clair/sombre).

## Contenu de l'overlay (v1)

1. **Jauges** : bloc 5h courant · 7 jours global · 7 jours Fable/Opus — chacune avec %, valeur, reset.
2. **MCPs** : nom, scope (global/projet), type (stdio/http/sse). Statique (pas de health-check en v1).
3. **Compte** : plan + tier, nom/email, organisation, modèle par défaut, version de Claude Code, activité du jour (messages, tokens).

## Hors scope v1 (backlog Roadmapped, déjà décidées avec Florian)

- **Notifications de seuil** (80 % / 95 % d'une jauge + badge tray) — plugin notification.
- **Raccourci clavier global** pour ouvrir/fermer l'overlay — plugin global-shortcut.
- **Lancement au démarrage** (login item) — plugin autostart.
- Idées ultérieures (proposées par l'agent, à trier) : stats par projet, historique graphique, health-check MCP, multi-comptes, distribution signée.

## Critères de done (v1)

1. `cargo test` vert : parsing JSONL (lignes valides, malformées, dédup) et calcul des fenêtres (bornes, familles de modèle) couverts.
2. L'app buildée s'affiche dans la barre de menu ; clic → overlay < 1 s avec les **vraies données de la machine** ; clic ailleurs → fermeture.
3. Les 3 sections affichent des données réelles cohérentes avec `/usage` (ordre de grandeur) et `~/.claude.json`.
4. Débrancher une source (renommer temporairement un fichier de config en test manuel) dégrade la section concernée sans casser le reste.

## Risques assumés

- Les formats de fichiers de Claude Code sont **non documentés et instables** : le collecteur doit être défensif (champs optionnels partout) ; une tâche de dette est ouverte pour suivre les changements de format.
- Les jauges sont des estimations : elles peuvent diverger de `/usage`, c'est affiché à l'utilisateur.
